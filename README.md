## 零、序曲——一杯咖啡与一条提示词

2026 年某个周末，我坐在电脑前，手里端着一杯咖啡，对着 Cursor 敲下了这样一条提示词：

> 万能的 AI，请给我写一个简单的微博。

AI 很快就给出了代码。一个 Express 服务，一个路由，返回"这是一个简单的微博"。架构干净利落——User 直接打 Node.js，Node.js 直接连 MySQL。没有缓存，没有消息队列，没有负载均衡。三层直连，多纯粹。

我当时想：**毕竟能有多少人用呢？**

每个后端工程师在最开始都是这么想的。从 1 个用户到 10 个、100 个，系统能扛。1000 个、10000 个……开始有点吃力了。等你反应过来的时候，用户数已经到了十万，而你的那条简单 SQL，正在扫遍整张百万行的表。

这大概就是 DDIA 这本书想要告诉我们的核心命题：**数据密集型系统的设计，不是"等出了问题再优化"，而是从一开始就要理解底层原理，在正确的时刻做正确的架构决策。**

今天我们用一个微薄的成长故事，把 DDIA 里最核心的几个章节串一遍。不是逐章逐节地念，而是跟着这个系统从零到分布式，看每一关我们遇到了什么问题，DDIA 又给了我们怎样的思维方式。

## 一、Day 1——用户越来越多了

### 从一条慢查询说起

tweets 表很简单：`id`、`user_id`、`content`、`create_time`，只给 `user_id` 建了一个单列索引。首页时间线的查询——取出每个用户最新的推文，按时间倒序排前 50 条——在数据量小的时候跑得很快。

但数据从 100 行到 100 万行，情况急剧恶化：

| 数据行数 | 100 | 1,000 | 10,000 | 100,000 | 1,000,000 |
|----------|-----|-------|---------|----------|-----------|
| 查询耗时 | 5ms | 50ms | 300ms | 1,200ms | 3,000ms |
| CPU 占用 | 10% | 20% | 50% | 80% | 100% |

然后你收到了告警——**首页打开失败，数据库连接超时**。

"我什么都没改啊……代码没动，配置没动……"

相信我，每一位后端工程师在事故前都说过这句话。

### EXPLAIN 揭开真相

把那条 SQL 扔进 EXPLAIN，真相大白：`type: ALL`，`rows: 1,000,000`，`Extra: Using filesort`。全表扫描。数据库必须把一百万行读出来逐行检查，再排序。这不是慢，这是在自杀。

### 为什么加了索引就快了？

加一条复合索引：

```sql
ALTER TABLE tweets ADD INDEX idx_user_time (user_id, create_time);
```

查询时间从 3000ms 直接干到 5ms——**600 倍提升**。

为什么？因为 B+Tree 的结构决定了，你不需要遍历整张表。`user_id` 在索引的第一层就能定位到目标范围，`create_time` 的排序已经在叶子节点的链表里了，范围扫描直接顺着链表走就行，连 filesort 都省了。

我们来算一笔账。一个 B+Tree 节点通常是 16KB（InnoDB 默认页大小），假设每条索引键占 24 字节加一个 8 字节的指针，也就是 32 字节走一个 entry，一个页能放大约 500 个 entry——这就是分支因子（fanout）。四层的 B+Tree 能存多少行？500⁴ ≈ 62 亿。也就是说，**在我们的场景里，最多 4 次磁盘读取就能定位到任意一行**。从一百万次磁盘读取变成四次，这就是 600 倍的来源。

### 三种存储引擎，三种取舍

DDIA 专门用了一章来讲存储引擎，核心是三种结构的对比：

- **哈希索引**：O(1) 点查极快，但完全不支持范围查询。适合 key-value 缓存场景，但微博时间线这种"按时间范围扫描"的需求直接排除。
- **B+Tree**：O(log n) 读写，叶子节点用双向链表串联，天然支持范围扫描。MySQL InnoDB 的默认选择，也是绝大多数 OLTP 场景的最优解。
- **LSM-Tree**：写入先到内存的 MemTable，满了再刷盘成 SSTable。写入极快（纯追加），但读取可能要合并多层文件。LevelDB、RocksDB、Cassandra 都在用。代价是读放大和后台 compaction 带来的资源抖动。

选存储引擎没有银弹，关键看你的读写模式是写多读少还是读多写少。

### 聚集索引 vs 非聚集索引

再深入一点。InnoDB 里，主键就是聚集索引——叶子节点直接存完整行数据。而二级索引（非聚集索引）的叶子节点存的是主键值。所以当你通过二级索引查一行数据，如果索引没有覆盖你需要的所有列，就要**回表**——先从二级索引找到主键，再从主键索引找到完整行。两次 B+Tree 查找。

覆盖索引就是绕过回表的技巧：把查询需要的所有列都放进索引里，这样索引本身就能回答查询，不需要再回到主键索引去取数据。代价是写入放大——你多建一个索引，每次 INSERT 就多一份 B+Tree 要更新，可能触发页分裂和树再平衡。InnoDB 建议每表 3-5 个索引，定期用 EXPLAIN 清理慢查询和无用索引。

到这里你应该已经感受到了——

> **索引不是优化。索引是生存条件。**

这不是夸张。在没有索引的世界里，数据库面对百万行数据就是一具步行 corpse。不是慢一点的问题，是根本不可用的问题。

## 二、用户突破一百万——单机撑不住了

### 从一台机到两台机

用户过了百万，单台 MySQL 的 CPU 已经 100% 了。读写全压在一台机器上，Web 服务到 MySQL 之间的那条线在告警图里永远是红色的。

第一反应是什么？"再买一台服务器。"

听起来简单。但数据怎么办？第二台机器上的数据和第一台不一样啊。

所以引入了**主从复制**：Leader 节点处理所有写入，通过 binlog 将变更同步到只读的 Replica 节点。应用层做读写分离——写走 Leader，读走 Replica：

```typescript
// 写入 → Leader
const tweet = await db.leader.tweets.create({
  data: { content: "Hello DDIA", userId: 42 }
});

// 读取 → Replica
const timeline = await db.replica.timeline.findMany({
  where: { userId: 42 },
  orderBy: { createdAt: 'desc' }
});
```

架构立刻从 User → MySQL (CPU 100%) 变成了 User → Leader (读写) → binlog → Replica (只读)。读的压力分走了，问题解决了吧？

### 复制滞后——你刚发的推呢？

问题来了。用户发了"Hello DDIA"，立刻刷新时间线——**Tweet Not Found**。为什么？因为 Replica 还在同步 binlog。等了大约 2 秒，数据终于一致了，推文出现了。

这就是异步复制的代价：**数据最终会一致，但不是立刻**。

你可能会说，那就用同步复制啊——Leader 等 Replica 确认了再返回成功。可以，但写入延迟直接翻倍，而且一旦 Replica 挂了，整个写入链路就堵死了。所以实际生产中最常用的是**半同步复制**：一个 Follower 同步（至少两台有数据），其余异步。这是安全性和性能之间最务实的平衡点。

### 复制滞后的三大异常

DDIA 把异步复制可能导致的不一致归纳为三种经典异常：

**Read-After-Write Consistency**——用户写入后立刻读，结果读到了旧数据。因为你命中了一个还没追上 binlog 的 Replica。解法：用户自己的写入走 Leader 读，或者对比时间戳判断 Replica 是否已追上。

**Monotonic Reads**——用户先读到新数据，再刷一次反而读到了旧数据，仿佛时光倒流。解法：基于用户 ID 哈希，让同一用户始终路由到同一个 Replica。

**Consistent Prefix Reads**——因果相关的消息顺序错乱，回复比原帖先出现。这在分区数据库里尤其常见。解法：确保因果相关的写入进入同一个分区。

这三种异常是分布系统实践中的高频坑。理解它们，你才能真正理解什么叫"最终一致性"——最终会一致，但在不一致的窗口里，什么怪事都可能发生。

> **复制让你睡得着觉，但复制滞后让你半夜惊醒。**

### 多主复制的冲突问题

如果你的系统需要跨地域部署——比如上海和新加坡各一个数据中心，两边都能写入——就要面对多主复制。两个 Leader 同时修改同一条数据，冲突不可避免。DDIA 介绍了几种冲突解决策略：

- **LWW (Last Write Wins)**：时间戳最新的为准，简单粗暴，可能丢数据。Cassandra 默认策略。
- **版本向量**：跟踪每个副本的版本号，检测并发冲突，交给应用层自定义合并逻辑。DynamoDB 使用。
- **CRDTs**：无冲突复制数据类型，数据结构本身保证合并后收敛。Redis CRDB 使用。

每种策略都是在一致性、复杂度和性能之间做取舍。没有万能答案，只有适合你场景的选择。

## 三、一个明星注册了——分区与热点

### 当数据大到一台机器放不下

某个早上你打开监控，Storage 使用量从 100GB 跳到了 500GB。原因很简单：一个拥有 1 亿粉丝的明星注册了你的微博。

数据继续膨胀——500GB → 1000GB → 2000GB。单机已经放不下了。

**分片（Partitioning / Sharding）** 是唯一的出路。最简单的哈希分片——取模：

```typescript
const shard = userId % 4;
db[shard].insert(record);
```

正常用户的数据均匀分布到 4 个 Shard，皆大欢喜。

但是 SuperStar 的 1 亿粉丝全部落入同一个 Shard——比如 Shard 2。这就是**热点分区（Hot Partition）**。数据倾斜了。一个 Shard 热到冒烟，其他三个闲得发慌。

> 分片不难，难的是热点。

### 两种分区策略

DDIA 讲了两种基本的分区方式：

**Key Range Partitioning**：按主键范围分区，A-C 到分片 1，D-F 到分片 2……优势是支持高效范围扫描，但新数据往往集中在最后一个分区，天然热点。

**Hash Partitioning**：对主键做哈希后分配分区。数据分布均匀，但范围查询需要扫描所有分区。实际生产中最常用的方案。一致性哈希在此基础上进一步优化了增减节点时的数据迁移量——将节点映射到哈希环上，只在相邻节点间迁移数据。Cassandra 和 Dynamo 都采用这种方式。

### 二级索引在分区世界里的麻烦

分区之后，二级索引怎么建？分两种：

**本地索引（Document-Partitioned）**：每个 Shard 各自维护自己的二级索引。写入只涉及单分区，快。但查询要发到所有分区做 scatter/gather，再合并结果。适合写多读少。

**全局索引（Term-Partitioned）**：二级索引本身也按 key 分区存储。查询只需命中少数分区，快。但写入一条数据可能要更新多个索引分区，涉及分布式事务，慢。适合读多写少。

选择哪种，取决于你的读写模式。

### 热点的缓解

对 SuperStar 这种极端热点，一个常用技巧是**给 key 加随机后缀**：`superstar#1`、`superstar#2`……读取时再聚合。增加了代码复杂度，但能有效分散负载。同时，像 Presto、ClickHouse 这样的 MPP 数据库还支持把单个大查询拆成多个子查询，在不同分区上并行执行。

分区还会带来一个运维问题：**再平衡（Rebalancing）**。当集群增减节点，数据要重新分配。固定分区数（比如预分 1000 个分区）、动态分区（大小超出阈值自动分裂）、节点比例分区（每节点固定分区数）——三种策略各有适用场景，Cassandra 用的是第三种。

## 四、流量洪峰——消息队列与流处理

### 一亿人同时在线

SuperStar 发了一条推文，1 亿粉丝同时拉取。我们第一版的代码是这么写的：

```typescript
await Promise.all(
  followers.map(f => sendNotification(f, tweet))
);
// 响应时间：100ms → 1s → 5s → 30s → Timeout 💥
```

同步 fan-out，1 亿条通知逐个推。服务端内存炸了，连接池耗尽了，客户端也等崩了。

**服务器炸了。同步 fan-out 不适用于高扇出场景。**

### Kafka 登场——削峰填谷

解法是异步化。把通知推送到 Kafka，生产者写入即返回：

```typescript
await kafka.publish('tweets', {
  userId: 'superstar',
  tweet: content,
  timestamp: Date.now(),
});
// 写入成功即返回——延迟 < 10ms ✅
```

消费者按自己的节奏消费：

```typescript
const consumer = kafka.consumer({ groupId: 'notif' });
await consumer.subscribe({ topic: 'tweets' });

while (true) {
  const { message } = await consumer.consume();
  await sendNotification(message.userId, message.tweet);
}
```

流量洪峰来了，Kafka 就是那个水库——先蓄水，再平稳泄洪。生产者不用等消费者处理完，消费者不会被洪峰击穿。**削峰填谷，保护下游。**

### AMQP vs Log-Based：两种消息哲学

这里值得展开说一下。消息系统本质上分两大流派：

**AMQP/JMS（传统）**：消息被消费后即删除。支持复杂的路由、优先级、TTL。RabbitMQ、ActiveMQ。适合 RPC 和任务分发。核心思维是：**消息是任务，处理完就扔掉**。

**Log-Based（日志式）**：消息持久化追加到日志中，消费者按 offset 读取，消息不删除（或按时间/大小保留）。Kafka、Pulsar。适合事件流处理。核心思维是：**消息是事实，永久记录可重放**。

这个区别决定了你能做什么。在 Log-Based 系统里，下游系统可以随时回到任意时间点重新消费数据——这就是 CDC（Change Data Capture）的基础。Debezium 监控数据库的 binlog，把每一次 INSERT/UPDATE/DELETE 变成事件流发到 Kafka，搜索索引、缓存、数据仓库各自订阅并更新。你再也不需要手动写双写了。

事件溯源（Event Sourcing）更进一步——根本不存储当前状态，只存储所有状态变更事件。当前状态 = 重放所有事件的结果。天然的审计日志，支持时间旅行。

### 流处理 vs 批处理

DDIA 还对比了两种数据处理范式：

- **批处理**：定期处理大量静止数据，T+1 模式。MapReduce、Spark。延迟高但吞吐高，适合报表和 ETL。
- **流处理**：持续处理实时到达的数据。Kafka Streams、Flink。延迟在毫秒到秒级，适合实时监控、风控、推荐。

Lambda 架构同时跑两层——批处理做基准，流处理做增量。但维护两套代码的复杂度很高。Kappa 架构主张：只用流处理，批处理不过是限定了时间窗口的流处理。简化了系统，但要求消息队列有足够的回溯能力。

### Exactly-Once 与窗口

最后提两个流处理里的硬核概念：

**Exactly-Once 语义**：消息恰好被处理一次。Kafka 通过事务 API 和幂等 Producer 支持。但要注意，"恰好一次"指的是效果上的恰好一次——内部可能重试，只是重试不会产生副作用。

**窗口类型**：滚动窗口（Tumbling）固定大小无重叠，每 5 分钟统计一次；跳跃窗口（Hopping）固定大小有重叠，每 1 分钟统计过去 5 分钟；会话窗口（Session）动态大小基于活动间隔，适合用户行为分析。

> **消息队列不是银弹，但它是你唯一的选择。**

## 五、微博开始赚钱——事务的代价

### 转账：一个经典的开场

微博开始赚钱了，用户充值余额，广告系统自动扣费。A 向 B 转账 ¥100：

```sql
-- 第 1 步：扣 A 的钱
UPDATE accounts SET balance = balance - 100 WHERE user = 'A';

-- 第 2 步：加 B 的钱
UPDATE accounts SET balance = balance + 100 WHERE user = 'B';
```

两条 UPDATE，看起来没问题。

然后——**断电！数据库崩溃！**

第 1 步执行了，第 2 步没执行。A 的余额变成了 ¥0（-100 已扣），B 的余额还是 ¥0（+100 没加上）。**¥100 凭空消失了。**

> 没有事务保护的数据库：钱会丢、数据会乱、用户会炸！

### 事务：要么全做，要么全不做

```sql
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE user = 'A';
  UPDATE accounts SET balance = balance + 100 WHERE user = 'B';
COMMIT;
-- 如果中途崩溃，数据库自动 ROLLBACK
-- A 的钱不会丢！
```

BEGIN 和 COMMIT 之间的操作是原子的。全成功则 COMMIT，任意失败则 ROLLBACK。

### WAL：崩溃恢复的秘密

那数据库怎么做到崩溃恢复的？靠 **Write-Ahead Log（预写日志）**：所有修改先写入 WAL 日志（txn=1, A-=100, B+=100, commit），然后再写入数据文件。每次事务提交时强制 `fsync` WAL 到磁盘。

崩溃恢复时，数据库读 WAL：已经 COMMIT 的事务重做（Redo），没有 COMMIT 的事务回滚（Undo）。这样就能保证 A 和 B 的余额永远一致。

实际生产中还有一个优化叫 **Group Commit**——把多个事务的 fsync 合并成一次，减少磁盘写次数，提升高并发写入吞吐。这也是为什么 WAL 顺序写入比随机写入快得多的原因之一。

### ACID：四个字母，四个承诺

- **A — Atomicity（原子性）**：这个词经常被误解为"同时发生"。其实它指的是可中止性——如果事务中途出了问题，所有已执行的操作必须被撤销。不是"同时"，是"要么全做，要么全不做"。
- **C — Consistency（一致性）**：这是应用层的责任，不是数据库的保证。数据库只管不违反约束（外键、唯一性），但"账户总金额守恒"这种业务规则，必须由开发者在事务逻辑里保证。
- **I — Isolation（隔离性）**：并发事务之间互不干扰。这是最常被妥协的属性——严格的隔离影响性能，所以实际中大家往往选择较低的隔离级别。
- **D — Durability（持久性）**：事务提交后，即使断电数据也不会丢。靠 WAL + fsync。但"持久性"是概率性的——磁盘也可能坏，所以生产环境要做 RAID、备份、异地容灾。

### 隔离级别：从 Read Uncommitted 到 Serializable

SQL 标准定义了四个隔离级别，从弱到强：Read Uncommitted → Read Committed → Snapshot Isolation (Repeatable Read) → Serializable。

每个级别都解决了上一级的问题，但又引入了新的坑。低于 Serializable 时，你可能遇到五种并发异常：

- **脏读（Dirty Read）**：读到另一个事务未提交的修改。Read Committed 及以上解决。
- **不可重复读（Non-Repeatable Read）**：同一事务两次读同一行结果不同。Snapshot Isolation 解决。
- **幻读（Phantom Read）**：同一事务两次范围查询返回不同行集合。需要谓词锁或索引范围锁解决。
- **丢失更新（Lost Update）**：两个事务同时读-改-写，后写入的覆盖先写入的。需要原子操作或显式锁。
- **写偏斜（Write Skew）**：两个事务读取相同数据后各自做决策并写入，Snapshot Isolation 无法防止。典型场景：医生值班系统——至少要有一人值班，但两个人同时看到"还有人在"就都请假了。需要 Serializable 才能彻底解决。

### 2PL vs SSI：两种实现可序列化的方式

**两阶段锁（2PL）**：事务持有读锁和写锁直到结束。强一致性，但吞吐量低、死锁频繁。MySQL 的 SERIALIZABLE 级别用的就是 2PL。

**可序列化快照隔离（SSI）**：基于 Snapshot Isolation，增加写偏斜冲突检测。事务提交时检查是否有冲突，有就回滚。无锁，乐观并发控制。PostgreSQL 9.1+ 的 SERIALIZABLE 用的就是 SSI。性能好很多，但冲突率高时回滚频繁。

### 分布式事务：2PC vs Saga

当事务跨多个节点，单机事务就不够了。

**两阶段提交（2PC）**：协调者先问所有参与者"你能提交吗？"（Prepare），全部同意后才发 Commit。任一节点故障则全部回滚。问题：同步阻塞、协调者单点、太重。XA 协议是标准实现。

**Saga 模式**：把大事务拆成多个本地事务序列，每个步骤有对应的补偿操作。失败了逆序执行补偿。异步、松耦合、最终一致性。适合长时间运行的业务流程——比如电商的"下单→扣库存→支付→发货"链路，任何一步失败都有对应的补偿动作（取消订单、回补库存、退款）。

> **没有事务的数据库不是数据库，是记事本。**

## 六、用户遍布全球——一致性的代价

### 光速限制了你

应用部署在上海，但用户在北京和新加坡同时访问。三大数据中心之间：

- 上海 → 北京：~40ms
- 北京 → 新加坡：~120ms
- 上海 → 新加坡：~150ms

物理距离等于网络延迟，这是光速决定的，没法优化。

### 消息顺序的崩塌

用户在北京发了"你好"，然后"吃了吗"，然后北京的另一个用户回复了。但"回复"从北京到上海只要 40ms，"吃了吗"从新加坡到上海要 120ms。结果上海的用户看到的顺序变成了：你好 → 回复 → 吃了吗。

**回复比原消息先出现。**

这就是最终一致性的日常。数据最终会一致，但在这段不一致的窗口里（大约 40~150ms），用户看到的就是乱序的消息。

### 三种一致性模型

DDIA 把一致性模型讲得很清楚，我这里重点对比三种：

**强一致性**（同步复制、2PC）：写入后立即对所有节点可见。无歧义，但延迟高。网络分区时系统不可用。

**最终一致性**（Cassandra、DynamoDB）：如果不继续写入，最终所有副本会收敛到相同值。高可用、低延迟，但可能读到旧数据。

**线性一致性**（etcd、ZooKeeper、Raft）：最强的单对象一致性模型——让多副本表现得像只有一个副本。如果操作 A 在操作 B 开始之前完成，所有节点都必须看到 A 发生在 B 之前。代价：网络延迟直接影响响应时间，网络分区时必须牺牲可用性。

一个常见的误解是把线性一致性和可序列化混为一谈。**它们是完全不同维度的保证**：线性一致性作用在单对象操作上（保证实时顺序），可序列化作用在多对象事务上（保证等价于某个串行执行顺序）。一个系统可以线性一致但不可序列化，也可以可序列化但非线性一致，两者都具备的叫严格可序列化（Strict Serializable）。

### CAP 的真相

最后说说 CAP 定理。它经常被简化成"三选二"，这其实是不准确的。CAP 的准确表述是：**当网络分区发生时，必须在一致性和可用性之间二选一**。注意前提——分区发生时。没有分区的时候，你可以同时拥有 C 和 A。

- CP 系统（etcd、ZooKeeper）：分区时牺牲可用性保持一致性。客户端请求可能被拒绝。
- AP 系统（Cassandra、DynamoDB）：分区时牺牲一致性保持可用性。返回可能过期的数据。

PACELC 定理是 CAP 的扩展，更贴近实际：**分区时在 A 和 C 之间选，无分区时在延迟（L）和一致性（C）之间选**。正常情况下你也得做选择——要线性一致性就意味着每次读都要等同步完成，延迟就上去了。

> **一致性是分布式系统最贵的奢侈品。**

## 七、最终 Boss——当所有节点都"正确"，但系统"错了"

### 双 11 零点：超卖

库存仅剩 1 件。双 11 零点整，User A 和 User B 同时抢购。两个节点各自检查本地库存——都看到 1 件——都允许下单。

**2 件售出，库存只有 1 件。超卖了。**

每个节点都"正确"——它看到的是自己的数据副本，库存确实是 1。但系统整体错了。这就是分布式一致性问题最尖锐的表现形式。

> 为什么每个节点都正确，结果却错了？
> 答案：缺乏**共识（Consensus）**。

### Raft 共识算法

DDIA 在第 9 章讲共识，Raft 是讲得最通透的算法之一。Raft 把共识分解为三个子问题：Leader 选举、日志复制、安全性。

**Phase 1: Leader 选举**

每个节点有一个随机超时时间（150-300ms）。超时后，节点从 Follower 变成 Candidate，Term 自增，向其他节点发送 RequestVote RPC。获得多数票就变成 Leader。Term（任期）单调递增，保证了不会出现两个合法 Leader 同时存在——也就是防止了脑裂。

**Phase 2: 日志复制（Log Replication）**

所有客户端请求都经过 Leader。Leader 把操作写入自己的日志，然后通过 AppendEntries RPC 复制到 Followers。多数节点确认后，Leader 提交（Commit）这条日志，然后通知客户端成功。已提交的条目保证最终被所有节点执行。

这里有一个关键的安全约束：**选举限制**——Candidate 的日志必须至少和多数节点一样新（比较最后日志的 term 和 index）。这保证了新 Leader 一定拥有所有已提交的日志条目。还有一个叫 **Figure 8 问题**的边界情况：Leader 不能提交旧 term 的条目，必须先提交一条当前 term 的条目来间接提交之前的日志。

### 成员变更：不停机升级集群

Raft 还解决了一个很实际的问题——集群配置变更（加减节点）期间如何保证不出现双 Leader。方案是 **Joint Consensus**：新旧配置共存的过渡期里，需要两个多数派（旧配置 + 新配置）都同意才能提交。保证过渡期间绝对不会出现两个 Leader。

### 三大共识算法对比

- **Paxos**：学术界经典，理论完备但难理解难实现。Google Chubby 使用。
- **Raft**：为可理解性设计，所有操作通过 Leader。etcd、TiKV、Consul 使用。
- **Zab**：ZooKeeper 定制，与 Raft 相似但选举细节不同。支持 FIFO 客户端顺序保证。

还有一个 FLP 不可能性定理值得了解：**在异步网络模型中，即使只有一个节点可能故障，也不存在总是终止的确定性共识算法**。听起来很绝望，但实际系统通过超时（引入随机化）绕过了这个理论限制。

## 八、终章——回顾：从三行代码到分布式系统

让我们回到最开始。

User → Node.js → MySQL。三行代码，三层架构。

然后：

- **流量来了**：加入 Load Balancer
- **缓存 + 消息队列**：Cache 和 Kafka 登场
- **分片扩展**：Sharding 应对数据膨胀
- **冗余副本**：Replication 保证可用性
- **保证正确**：Transactions 让数据不丢不乱
- **达成共识**：Consensus 在分布式环境下做出一致决策

系统一步步从三层直连走向了完整的分布式架构。**每一次演进，都是因为原来的方案扛不住了。每一次选择，都是在不同的 trade-off 之间做取舍。**

DDIA 这本书的核心不是教你某个具体的技术选型，而是帮你建立一种思维方式：**面对数据密集型系统的问题时，理解底层原理，理清 trade-off，在正确的时刻做正确的决策。**

没有银弹。索引有代价，复制有滞后，分片有热点，事务有性能损耗，一致性有延迟成本，共识有复杂度。每一个"解决方案"都同时引入了新的"问题"。但正是这些权衡和选择，定义了你的系统长什么样。
